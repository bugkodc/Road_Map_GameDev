# Dead Code

> 📖 **Source:** [Refactoring.Guru — Dead Code](https://refactoring.guru/refactoring/smells/dispensables) | Author: Alexander Shvets

## 📋 Signs & Symptoms

**Dead Code** is any piece of code (a variable, parameter, method, property, class, or file) that is **never called, used, or executed** throughout the entire lifetime of the application.

> 🗑️ **The supreme rule:** If code is never used, the one and only best solution is to **DELETE IT**. Don't comment it out to save for the future. You can always find it again in your Version Control (Git) history.

## ❓ Reasons for the Problem

- **Constantly changing requirements**: Feature A is removed or replaced by feature B; the programmer writes the new code but forgets to delete the code that supported the old feature.
- **Saving it for the future**: Thinking "this code might come in handy later," so it gets commented out or left in place.
- **Boilerplate (auto-generated functions)**: IDEs or game engines auto-generate empty callback functions when a new file is created, and the programmer is too lazy to delete them.

## 💊 Treatment

1. **Delete**: Mercilessly delete unused files, classes, methods, or variables.
2. **Use IDE static analysis tools**: Modern IDEs like Rider, Visual Studio, and VS Code all warn you by greying out code with no references (0 references). Trust them and clean it up.
3. **Remove Unused Boilerplate**: Delete the engine's default empty functions if you haven't written anything in them.

## ✅ Payoff

- **Reduced cognitive load**: A lean codebase helps the programmer focus on the code that actually runs.
- **Faster compile time**: Less code means faster game builds.
- **Performance optimization**: Avoids pointless implicit calls from the engine.

## 🎮 In Game Dev

In Unity, Dead Code not only clutters the codebase but also directly **wastes CPU resources**:

### ❌ Example before refactoring:
A MonoBehaviour script with empty functions and an old debug variable:
```csharp
public class PlayerController : MonoBehaviour
{
    // Old variable used for a shooting prototype that has now been dropped
    private float oldFireRate = 0.5f;

    // Empty Start function generated by Unity
    void Start()
    {
        
    }

    // Empty Update function generated by Unity
    void Update()
    {
        
    }
}
```
**The harm in Unity:** 
- The `oldFireRate` variable wastes memory for no reason.
- **Most dangerous of all:** Even when `Start()` and `Update()` are empty, Unity still has to make bridge calls (C++ to C# bridge calls) for each of these MonoBehaviours every frame. If you have 1,000 monsters carrying a script with an empty `Update()`, your game will suffer a significant FPS drop for absolutely no benefit!

### ✅ Solution after refactoring:
Delete everything that isn't used, completely:
```csharp
public class PlayerController : MonoBehaviour
{
    // Clean — no redundant variables, no empty callback functions!
}
```
